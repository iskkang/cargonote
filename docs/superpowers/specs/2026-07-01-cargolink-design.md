# CargoLink — Phase 1 설계 스펙 (내부용 "이메일 킬러")

> **상태:** 검토용 초안 · **작성일:** 2026-07-01 · **대상:** MTL 내부 1차 릴리스
> **한 줄 정의:** 사무실이 작업을 지시하면 → 현장은 무설치 링크로 찍기만 → 컨테이너별로 자동 정리 → 링크 한 줄로 공유.
> **PRD와의 관계:** 이 스펙은 PRD v1.2의 방향을 **내부용 1차 릴리스로 좁힌 것**이다. V0 / v1.1 / v1.2는 여기까지 온 논리 사슬이며, 본 문서가 Phase 1의 확정 스펙이다.

---

## 1. 배경 & 문제

**현재 MTL 워크플로우 (교체 대상):**
1. 사무실 직원이 현장 작업자에게 "오늘 이 화물을 이 컨테이너에 적입한다"고 사전 통보.
2. 작업 당일, 작업자가 작업 사진을 찍어 **이메일로 전송**.
3. 사무실이 파일을 **다운로드 → 컨테이너별로 수동 정리 → 해외 파트너·지사에 재발송**.

**비용:** 작업당 20~30분 증발, 이메일 용량 초과, 로컬 저장·재발송 반복, 누락·속인화.

**Phase 1이 푸는 것:** 위 과정을 **URL 한 줄**로 대체 → **업무 단순화 + 이메일 용량 감소 + 업무 누수 방지.**

---

## 2. 범위 결정 — Approach A "이메일 킬러"

MTL 자체 작업장에서 **채택을 증명하는 가장 얇은 제품**(Gate ①). 파리티·AI·과금은 채택 증명 후에 얹는다. 스키마는 확장 대비해 미리 열어두어 재작업을 없앤다.

| Phase 1 (만든다) | 의도적으로 뺌 (Phase 2+) |
|---|---|
| 무설치 웹 캡처(가이드 촬영) | AI 사진유형 태깅 / 데미지 판정 |
| 작업 지시·배정(관리자 웹) | OCR(컨테이너/Seal 자동입력) |
| 컨테이너별 자동 정리 + 갤러리 링크 | PDF 리포트 |
| 누락 체크(가이드 촬영, AI 없음) | 멀티테넌시(org RLS) / 과금 |
| 증빙 무결성(원본 불변·해시·GPS·audit) | RN 네이티브 앱(상시 오프라인 작업자용) |
| 캐리어별 템플릿(TSR / TCR) | 작업유형 확장(항공·차량·택배 등) |

---

## 3. 목표 / 비목표

**Phase 1 목표**
1. 사진 → 파트너 공유 시간을 "분 단위 수작업 → 링크 한 줄"로 단축.
2. 사진이 **컨테이너별로 자동 그룹핑**되어 파트너가 무설치·무로그인으로 열람.
3. 캐리어(TSR/TCR)가 요구하는 **필수 사진 누락을 가이드 촬영으로 방지**.
4. **MTL 자체 작업장에서 계속 쓰이는가**(채택 검증, Gate ①).

**비목표 (Phase 1)** — AI, OCR, PDF, 멀티테넌시/과금, 네이티브 앱, 치수 구조화 입력, 카카오 알림톡 자동발송, TMS/WMS·SSO·보험사 연동.

## 4. 페르소나 & 역할

| 역할 | 클라이언트 | 하는 일 | 인증 |
|---|---|---|---|
| **사무실(관리자)** | 웹 콘솔 (데스크톱) | 작업지시 생성·링크 발송·검토·**발행** | Supabase Auth 로그인 |
| **현장 작업자** | 무설치 웹(모바일 브라우저) | 토큰 링크 진입 → 가이드대로 촬영 → 제출 | **없음**(토큰 링크) |
| **수신자(해외 파트너·지사)** | 무설치 웹 | 발행된 갤러리 링크 **열람만** | **없음**(토큰 링크) |

- **작업자는 스스로 작업을 시작하지 않는다.** 항상 관리자가 콘솔에서 작업지시를 만든다.
- **최종 발행 게이트는 관리자 전용.** 작업자 제출 = 관리자에게 보고까지이며, 곧바로 파트너에게 공개되지 않는다.

## 5. 핵심 흐름 (7단계)

1. **관리자**가 콘솔에서 작업지시 생성 — 거래처(파트너/지사) · **템플릿(TSR/TCR)** · 컨테이너 번호(1개 이상) · 부킹/Ref · 작업일 · 담당 작업자(이름/연락처). 필수 사진 목록은 템플릿에서 자동.
2. 시스템이 **작업자용 토큰 링크** 발급 → 관리자가 SMS/메신저로 작업자에게 전송(공유시트/링크복사).
3. **작업자**가 링크 클릭(무설치) → 해당 컨테이너의 **필수 사진 체크리스트** 표시 → 슬롯별 촬영(연속 촬영, 자동 압축 + WebP 표시용·썸네일 생성, 촬영시각·GPS 기록) → 찍으면 **자동 체크(가이드 촬영)** → 누락 경고 → **제출**.
4. 사진은 자동으로 그 작업지시·컨테이너에 **귀속**(번호를 이미 아니 OCR 불필요). 오프라인이면 로컬 큐(IndexedDB) → 복구 시 자동 업로드(중복 방지).
5. **관리자 콘솔에 실시간 반영** → 관리자가 컨테이너별로 검토(누락/흐림 확인) → **"검토 & 발행"**.
6. 발행 시: **수신자용 갤러리 링크** 생성(컨테이너별 정리 + OG 미리보기 카드), 그 시점 **사진 버전 고정**(증빙). 관리자가 링크를 파트너·지사에 공유.
7. **수신자**가 링크 열람(무설치·무로그인): 컨테이너별 사진 갤러리 + 기본정보 + 작업자 메모. → **이메일 워크플로우 대체 완료.**

**설계 원칙:** 사람 게이트 = "발행" 1개 · 작업자는 태그를 안 고름(목록대로 촬영만) · 컨테이너 번호는 작업지시에서 옴.

## 6. 작업유형 템플릿 (캐리어/루트 구동)

**핵심 통찰:** 필수 사진은 임의 목록이 아니라 **캐리어·세관이 강제하는 규격**이다. 빠지면 실제 손해(철도 적재 거부, 한국 반송, 추가 비용)가 발생한다. 따라서 템플릿은 "작업유형"이 아니라 **"캐리어/루트별 사진 스펙"**이다. Phase 1은 MTL 실무인 **TSR·TCR 2종을 시드**한다. (이 자체가 ECOYA 등 한국 일반 툴에 없는 차별화 자산.)

**철칙 (두 템플릿 공통):** 모든 사진(CSC 제외)에 **컨테이너 번호가 보이게** 촬영. 쇼링·고박은 단단히 고정 후 촬영.

### 6-A. TSR (시베리아 횡단철도 · FESCO 선사) — 최소 7장 + CSC

| 슬롯 | 촬영 지시 |
|---|---|
| 빈 컨테이너 | 번호 보이게 |
| 절반 적재 | 번호 보이게 |
| 만재(FULL) | 번호 보이게 |
| 쇼링·고박 후 | 번호 보이게, 단단히 고정 |
| 한쪽 문 닫힘 | 번호 전체 보이게 |
| 봉인 완료(양쪽 문) | **말뚝씰 · 오른쪽 문, 왼쪽 레바** |
| 씰 근접 | 씰 번호 판독 가능하게 |
| CSC 명판 | (번호 규칙 예외) |

- `warning_text`: "필수 7장 + CSC · 누락 시 러시아 철도 적재 거부 → 재마킹까지 지연 + 보관·작업료."
- 참고(FESCO 마킹 공지): 러시아 철도청 ISO 6346 외부 마킹 단속 강화 — 마킹 불량 시 철도 거부. 이 사진 세트가 마킹 적합 증빙 역할.

### 6-B. TCR (중국 횡단철도 · 중국 세관) — 최소 8장

| 슬롯 | 촬영 지시 |
|---|---|
| 빈 컨테이너 | 번호 보이게 |
| 절반 적재 | 번호 보이게 |
| 만재(FULL) | 번호 보이게 |
| 쇼링·고박 후 | 번호 보이게, 단단히 고정 |
| 한쪽 문 닫힘 | 번호 전체 보이게 |
| 봉인 완료(양쪽 문) | 씰 봉인 |
| 씰 근접 | 씰 번호 판독 가능하게 |
| CSC 명판 | 필수(번호 규칙 예외) |

- `warning_text`: "필수 8장 · 중국 세관 제출 필수, 미충족 시 한국으로 반송될 수 있음."

### 6-C. 템플릿 간 분리 지점 (데이터로 표현)
`min_count` · 슬롯별 `instruction` · **씰 규칙(종류/위치)** · **경고문(`warning_text`)** · `carrier`/`route`. 사진 세트는 ~공통, 이 필드들만 템플릿마다 다름.

## 7. 데이터 모델

정리·귀속 단위 = **컨테이너**. `work_order → containers` 는 1:N (1화주-N컨, 배치, LCL 커버). LCL은 컨테이너 사진을 통째로 파트너에게 보내므로 **화주(shipper)는 별도 테이블 없이 작업지시의 라벨/메모**로 충분.

```
customers (파트너/지사 = 수신처)
  └─ work_orders (작업지시)  ─── work_type_templates (TSR/TCR: 앵커·필수사진·규칙·경고)
        └─ containers (컨테이너 = 정리·귀속 단위, worker_memo 포함)
              └─ photos (원본·썸네일·해시·촬영시각·GPS·slot key·버전)
  share_links   (worker 촬영링크 / viewer 갤러리링크, 각 토큰)
  publications  (발행 스냅샷 = 사진 버전 고정, photo_manifest+hash)
  audit_logs    (누가·언제·무엇을)
  users         (관리자 = Supabase Auth. 작업자·수신자는 무계정 토큰)
```

| 테이블 | 핵심 컬럼 |
|---|---|
| `customers` | name, contact, notes |
| `work_type_templates` | name, carrier, route, anchor_type, **required_photos(jsonb: {key,label,instruction,required})**, rules(jsonb: container_no_visible, seal_type, seal_position), min_count, warning_text, `ocr_targets`·`tag_set`(빈칸) |
| `work_orders` | customer_id, template_id, work_date, status(`draft→sent→in_progress→submitted→published`), assignee_name/contact, shipper_label(nullable), created_by |
| `containers` | work_order_id, container_no, `seal_no`(nullable), **worker_memo(free text: 화물·PLT 사이즈)** |
| `photos` | container_id, slot_key(nullable=기타), original_path, thumb_path, **file_hash·captured_at·gps_lat/lng**, byte_size, status(uploaded/soft_deleted) |
| `share_links` | token, kind(worker/viewer), revoked, `expires_at`·`password`(nullable, P2) |
| `publications` | work_order_id, published_at/by, **photo_manifest(jsonb: [{photo_id,hash,container_id,slot_key}])**, viewer_token |
| `audit_logs` | entity, entity_id, action, actor, ts, detail(jsonb) |
| `users` | (Supabase Auth) 관리자 |

**미리 비워두는 자리(Phase 2 컬럼/테이블 추가만):** `photo_ai_labels`·`model_feedback`(AI) / `reports`(PDF) / `org_id`·roles(멀티테넌시) / share_link 만료·비번 / `seal_no` 구조화.

**누락 체크는 파생:** 필수 슬롯 중 `photos.slot_key`가 채워진 = 완료, 빈 = 누락.

3. **수신자 갤러리 (H3, 무로그인)** — 컨테이너별 정리된 사진(그리드 썸네일 → 탭 시 **표시용 압축본, 번호·씰 판독 가능**) + 기본정보 + 메모, OG 미리보기 카드, 체인오브커스터디(증빙) 표시.

## 9. 기술 스택

- **메인 클라이언트 = 웹 PWA (무설치).** 작업자가 정직원(회사폰)·정직원(개인폰)·외주 인력이 **섞여 있어** "전원 앱 설치"를 전제할 수 없음. 작업자 촬영 + 수신자 갤러리 모두 웹.
  - 카메라 `getUserMedia` / `<input capture>`, 오프라인 큐 IndexedDB + Service Worker, 공유 `navigator.share`, 진입 = 토큰 링크(무설치·무로그인).
- **관리자 웹 콘솔:** React + Vite (기존 Logisight/MTL Link 스택 정합).
- **백엔드/DB/Auth/Storage:** Supabase.
  - Postgres. 관리자 = Supabase Auth. 작업자·수신자 = **토큰 기반 접근**(Edge Function 또는 토큰 검증 RLS 정책, 사용자 인증 아님).
  - Storage 버킷 분리: `originals`(증빙 보존·불변) / `display`(표시용 압축 WebP — **번호·씰 판독 가능한 해상도**) / `thumbnails`(그리드). **링크엔 원본 대신 display/thumbnail만 노출 → egress 최소화**, 원본은 audit 전용. 접근 레이어 추상화(추후 R2 이전이 config 한 줄).
  - Phase 1은 단일 조직이라 org RLS는 자리만(비활성). 
- **OCR/AI 없음** → 비전/OCR 인프라·비용 Phase 1 제로.

## 13. 로드맵 & 게이트

1. **Phase 1 개발** — 무설치 웹 캡처 + 작업배정 + 컨테이너/작업 모델 + TSR/TCR 템플릿 + 웹 콘솔 + 갤러리 링크 + 오프라인 + 증빙. (착수 전: 파일럿 작업장 지정 + baseline 측정, iOS 1주차 검증.)

4. **Gate ②** — 외부 유료 파일럿 1곳.
